# 01 - RAG 是什么？为什么档案系统需要它

## 大模型的根本缺陷

你用过 ChatGPT，它很聪明，但有一个致命问题：

> **它只知道训练数据里的东西，不知道你们单位的档案。**

比如你问它："2023年3月5日，XX市政府关于城市改造的批复文件在哪里？"  
它只会胡编（这叫"幻觉"，hallucination），因为它的训练数据里根本没有这份文件。

## RAG：让大模型看到你的数据

RAG = **Retrieval-Augmented Generation**（检索增强生成）

思路很直接，分三步：

```
用户提问
   ↓
① 检索（Retrieval）：在档案库里搜索最相关的文件片段
   ↓
② 增强（Augmented）：把这些片段塞进给大模型的提示词里
   ↓
③ 生成（Generation）：大模型基于这些真实内容生成回答
```

用一个比喻理解：

| 没有 RAG | 有 RAG |
|---------|--------|
| 大模型像一个只读过教科书的学生 | 大模型像一个手边有完整档案库的助手 |
| 靠记忆回答，可能记错 | 先查阅相关文件，再基于文件回答 |
| 无法获取你的私有数据 | 可以检索你的私有档案 |

## 在档案系统里的具体工作流

```
档案入库阶段（提前做一次）:
  1. 档案文件（PDF/Word/扫描件）
       ↓ 文本提取
  2. 纯文本内容
       ↓ 切块（每块约 500 字）
  3. 文本片段列表
       ↓ Embedding 模型向量化
  4. 向量（一组数字，如 [0.12, -0.34, 0.89, ...]）
       ↓ 存入 pgvector
  5. 向量数据库（搜索时使用）

用户提问阶段（实时）:
  用户：这份批复文件的依据是什么？
       ↓ 同样的 Embedding 模型向量化
  问题向量 → pgvector 相似度搜索 → 找到最相关的 3-5 个文件片段
       ↓ 组装提示词
  "根据以下档案内容回答问题：\n[片段1]\n[片段2]\n...\n问题：这份批复的依据是什么？"
       ↓ 发给大模型（Ollama/qwen2.5）
  大模型生成回答（有据可查，不乱编）
```

## 关键概念：向量（Embedding）

这是 RAG 的核心技术点，下一篇文档 `02-vector-embeddings.md` 详细讲解。

简单理解：向量是一种**把文字转换成数字的方式**，语义相近的文字，转换后的数字也相近。

```
"关于城市改造的批复" → [0.12, 0.89, -0.34, ...]
"市政建设审批文件"   → [0.11, 0.87, -0.31, ...]   ← 很接近，因为语义相似
"今天天气真好"       → [0.98, -0.12, 0.45, ...]   ← 差很远
```

所以搜索不是关键词匹配，而是**语义相似度搜索**——即使用户没用完全一样的词，也能找到相关内容。

## 为什么档案系统特别适合 RAG

1. **数据私密性**：档案不能上传到 OpenAI 云端，Ollama 本地部署完美解决
2. **精确引用需求**：档案工作要求有据可查，RAG 能展示"引用来源是哪份文件"
3. **专业术语多**：通用搜索引擎不懂"全宗"、"案卷"、"SIP"，向量语义搜索天然适合
4. **跨文档关联**：一个问题可能涉及多份档案，RAG 能同时检索多个来源

## SAMS 中的 RAG 架构

```
档案员上传文件
      ↓
[Dify 知识库管理]     ← 处理文档解析、切块、向量化，有 UI 操作，不用写代码
      ↓
[pgvector 向量库]     ← 存在现有的 PostgreSQL 里，无需新数据库
      ↓
[Ollama - nomic-embed-text]  ← 本地 Embedding 模型，文字→向量
      ↓
[Ollama - qwen2.5]   ← 本地对话模型，生成最终回答
      ↑
[FastAPI 代理层]      ← 处理 SAMS 的认证、审计日志、租户隔离
      ↑
[Nuxt + @nlux/vue]  ← 用户看到的聊天界面
```
