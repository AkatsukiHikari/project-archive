# 02 - 向量与语义检索原理

## 为什么关键词搜索不够用

传统的档案检索用关键词：用户输入"城市改造"，系统找包含这三个字的文件。

**问题**：

- 用户输入"市政建设"，找不到写着"城市改造"的文件（同义词问题）
- 用户输入"拆迁补偿政策"，找不到相关的城市更新批复（上下位概念问题）
- 用户用了不同表述，就找不到（档案系统用语非常不统一）

**解决方案**：把文字变成向量，用"语义距离"而不是"字面匹配"来检索。

## 向量是什么

向量就是**一组有序的数字**。

Embedding 模型（如 nomic-embed-text）的工作是：
> 把任意一段文字，转换成一个固定长度的数字数组（768 个数字）

```python
"关于城市改造项目的批复文件"  →  [0.12, -0.34, 0.89, 0.45, ..., 0.23]  # 768个数字
"市政建设审批意见书"          →  [0.11, -0.32, 0.87, 0.43, ..., 0.21]  # 768个数字
"今天食堂的午饭很好吃"        →  [0.98, 0.76, -0.12, 0.01, ..., 0.55]  # 768个数字
```

前两句"语义相近"，所以它们的数字很接近。最后一句完全不相关，数字差距很大。

## 相似度计算：余弦相似度

判断两段文字"有多像"，用**余弦相似度**：

```
相似度 = 两个向量的夹角余弦值
  = 1.0  → 完全相同
  = 0.9+ → 高度相似（同一个意思，不同说法）
  = 0.5  → 有一定关联
  = 0.0  → 完全不相关
```

你不需要懂数学公式，只需要记住：**pgvector 会自动帮你算这个值，并按相似度排序**。

## pgvector：PostgreSQL 的向量插件

SAMS 用的是 `pgvector/pgvector:pg17` 镜像，这个镜像在标准 PostgreSQL 基础上加了向量支持。

在数据库里，一个向量字段长这样：

```sql
-- 档案文本片段表（Dify 会自动创建类似的表）
CREATE TABLE archive_embeddings (
    id          UUID PRIMARY KEY,
    content     TEXT,              -- 原始文本片段
    embedding   vector(768),       -- 向量（768维，对应 nomic-embed-text）
    source_doc  TEXT,              -- 来源文件名
    metadata    JSONB              -- 其他元数据
);

-- 创建向量索引（加速搜索，类似普通索引）
CREATE INDEX ON archive_embeddings USING ivfflat (embedding vector_cosine_ops);
```

**搜索时**（找和用户问题最相关的 5 个片段）：

```sql
SELECT content, source_doc,
       1 - (embedding <=> $1) AS similarity   -- <=> 是余弦距离运算符
FROM archive_embeddings
ORDER BY embedding <=> $1                      -- 按相似度升序（距离越小越相似）
LIMIT 5;
-- $1 是用户问题的向量
```

## 文档切块（Chunking）：为什么要切？

一份完整的档案可能有几十页，不能把整份文件做成一个向量，原因：

1. **精度问题**：向量代表整份文件的"平均语义"，检索时会失去细节
2. **Token 限制**：大模型的上下文窗口有限，不能把几十页全塞进去
3. **相关性问题**：用户问的可能只是文件里某一段，其他内容是噪音

**切块策略**（Dify 已内置，可在界面配置）：

```
原始文档（10页PDF）
    ↓ 按段落/句子分割
片段1: "第一条 根据《档案法》第XX条规定..."（500字）
片段2: "第二条 审批程序如下：..."（500字）
片段3: "第三条 经费来源..."（500字）
    ...
每个片段独立向量化，存入数据库
```

**重叠（Overlap）**：相邻片段之间保留约 50-100 字的重叠，防止一个完整句子被切断在两个片段里：

```
片段1: [... AAAAAAA BBBBB]
片段2:          [BBBBB CCCCCCC ...]
                 ↑ 重叠部分
```

## Embedding 模型选择：nomic-embed-text

SAMS 选择 `nomic-embed-text` 作为向量化模型，原因：

| 特性 | 说明 |
|------|------|
| 本地运行 | 通过 Ollama 离线运行，数据不出网 |
| 中文支持 | 对中文语义理解较好 |
| 维度 768 | 精度和性能的平衡点 |
| 速度快 | 相比大语言模型，embedding 模型很轻量 |
| 免费开源 | MIT 许可 |

**在 Ollama 中下载**：

```bash
ollama pull nomic-embed-text
```

## 对话模型选择：qwen2.5

RAG 的最后一步是让大模型"读懂"检索到的片段并生成回答，选 `qwen2.5` 原因：

| 特性 | 说明 |
|------|------|
| 阿里开源 | 对中文支持最好的开源模型之一 |
| 档案术语 | 训练数据包含大量中文政府文件 |
| 7B/14B 可选 | 普通 CPU 机器可跑 7B，有独显可跑 14B |
| 指令遵循好 | 能严格按照格式要求回答，减少"乱说" |

**在 Ollama 中下载**：

```bash
# 7B 版（CPU友好，约4GB内存）
ollama pull qwen2.5:7b

# 14B 版（效果更好，需约8GB内存）
ollama pull qwen2.5:14b
```

## 完整流程回顾

```
档案入库时:
  文档 → [nomic-embed-text] → 向量 → pgvector

用户提问时:
  问题 → [nomic-embed-text] → 问题向量
       → pgvector 余弦相似度搜索
       → 返回最相关 5 个文本片段

  组装提示词:
  "你是档案助手，请根据以下档案内容回答问题，不要编造。
   档案内容：
   【片段1】...
   【片段2】...
   问题：XXX"

  → [qwen2.5] → 最终回答（有文件依据）
```
